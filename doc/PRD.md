# 织策 AI (WeaveQA) MVP 产品需求文档 (PRD)

## 1. 产品定位与目标

**织策 AI (WeaveQA)** 是一款面向企业内部 QA 团队的**智能上下文增强型测试用例生成与评审工作台**。MVP 阶段聚焦验证“历史 PRD + Bug 知识库 RAG 是否能提升测试用例覆盖率、降低遗漏率”。飞书/Figma 真实接入、智能 Diff、脑图评审作为后续阶段能力预留。

---

## 2. 核心用户旅程 (User Workflow)

1. **构建资产：** QA 导入历史 PRD、历史 Bug 或混合文本资产，系统自动切分并建立可检索索引。
2. **新需输入：** 测试人员输入本次需求文本。
3. **召回调试：** 系统展示与当前需求相关的历史上下文片段。
4. **RAG 生成：** 系统生成结构化测试用例，并为每条用例附加来源引用和风险依据。
5. **质量评审：** QA 对比无知识库、仅 PRD、PRD + Bug 三种模式的覆盖率、遗漏风险和 Bug 回归点。

---

## 3. 详细功能需求

### 3.1 知识空间管理 (Knowledge Base)

* **【手动导入】** MVP 支持 PRD、Bug、混合文本三类知识资产导入。
* **【层级切分引擎】** 后端执行 Parent-Child 风格切分，子分块用于检索，父文本用于上下文引用。
* **【召回调试】** 前端展示检索片段、相似度、资产类型和来源 ID。

### 3.2 智能用例生成与路由控制 (AI Engine)

* **【生成模式】** 支持无知识库、仅 PRD、PRD + Bug 三种生成模式。
* **【结构化输出】** 用例字段包括模块、标题、前置条件、步骤、预期结果、优先级、关联风险、RAG 引用来源。
* **【模型路由预留】** 前端展示模型配置入口，后端保留 OpenAI Compatible 路由配置结构。
* **【UI 上下文融合预留】** Figma MCP 作为 Phase 3 能力，不纳入 MVP 交付范围。

### 3.3 变更管理与智能 Diff 评审 (Core Value)

* **【质量评审】** 记录覆盖评分、遗漏风险数量、Bug 回归点数量和采纳用例数。
* **【对比指标】** 使用同一需求对比三种生成模式，验证知识库是否带来覆盖提升。
* **【Diff 预留】** 需求变更 Diff、采纳/驳回、脑图联动放入 Phase 4/5。



---

## 4. 非功能性需求

* **安全性：** MVP 不保存真实 API Key；后续企业化阶段要求密钥加密和统一代理。
* **性能：** MVP 以质量验证优先；单次 RAG 生成应在 3 秒内返回。
* **可追溯性：** 每条 RAG 增强用例必须包含来源引用。
* **视觉风格：** 参考 Claude/Anthropic 的暖色画布、克制按钮、深色上下文面板和编辑感排版，但不照搬品牌元素。

# 系统技术架构图与技术栈规范

系统采用前后端分离架构。React 承载工作台界面，FastAPI 提供知识导入、检索、生成和评审 API。PostgreSQL 与 Qdrant 负责知识资产持久化与向量检索。

### 技术栈选型

* **前端 (Frontend):** React, Vite, CSS Variables, lucide-react。
* **后端 (Backend):** Python 3.11, FastAPI, Uvicorn, Pydantic。
* **中间件 & 数据库:** PostgreSQL 16 与 Qdrant 通过 Docker Compose 预留。

# MVP (最小可行性产品) 四周实施路线图

为了能够快速看到效果并向团队交付价值，建议将开发周期浓缩在 4 周内完成：

### 第 1 周：基础设施与工作台 (Foundation)

* **目标：** 跑通 React + FastAPI 可演示链路。
* **后端任务：** 提供健康检查、模型配置结构、知识导入 API。
* **前端任务：** 完成 RAG 工作台、知识导入、需求输入、模型入口和下一阶段路线图区块。

### 第 2 周：知识切分与召回 (Data Ingestion)

* **目标：** 让系统可以检索历史 PRD 与 Bug。
* **后端任务：** 实现 Parent-Child 风格切分、向量化、相似度召回。
* **前端任务：** 开发召回调试面板，展示命中片段、相似度和来源。

### 第 3 周：RAG 增强用例生成 (AI & Logic)

* **目标：** 生成带来源引用和风险依据的结构化测试用例。
* **后端任务：** 实现三种生成模式、用例结构化输出、覆盖评分和遗漏风险计算。
* **前端任务：** 展示生成用例表、引用来源和质量指标。



### 第 4 周：质量评审闭环 (UI & Integration)

* **目标：** 完成可演示的 RAG 价值验证闭环。
* **前端任务：** 保存覆盖评分、遗漏风险、Bug 回归点和采纳用例数。
* **验收方式：** 使用同一需求对比无知识库、仅 PRD、PRD + Bug 三种模式，确认 PRD + Bug 生成覆盖更完整且引用可追溯。

---

# 下一阶段目标

* **Phase 2：** 接入真实飞书文档同步，支持文件夹绑定、手动同步、Webhook 变更监听，并将知识资产持久化至 PostgreSQL/Qdrant。
* **Phase 3：** 接入 Figma MCP，将 UI 组件树、输入限制、按钮跳转关系纳入生成上下文。
* **Phase 4：** 实现需求变更 Diff，输出 `added`、`modified`、`deleted`、`unchanged` 状态，并支持人工采纳/驳回。
* **Phase 5：** 完善 MeterSphere 风格双视图，列表 Diff + React Flow 脑图联动评审。
* **Phase 6：** 补齐企业化能力，包括权限、审计、API Key 加密、统一代理、导出到测试平台。
